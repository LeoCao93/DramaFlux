# DramaFlux 根文档重构设计

## 背景

当前根目录 `README.md` 与 `DEPLOYMENT.md` 存在以下问题：

- 根 README 仍使用旧项目名，未统一为 `DramaFlux`
- README 信息层次过薄，不适合作为多人协作入口
- 部分文档存在过时描述、重复描述，或容易被误读为默认事实的本机绝对路径
- README 与部署文档的职责边界不够清晰
- 文档视觉层次较弱，缺少工程协作项目应有的结构化呈现

本次工作目标是重构这两份根文档，使其适合后续多人协作，并保持内容真实、清晰、易维护。

## 目标

重写根目录 `README.md` 与 `DEPLOYMENT.md`，建立清晰分工：

- `README.md` 作为项目门面与协作入口
- `DEPLOYMENT.md` 作为完整部署、启动、验证与排障手册

## 非目标

本次不包含以下内容：

- 不新增 logo、banner、PNG、SVG 等设计资产
- 不修改服务代码、接口实现或脚本行为
- 不重写 `services/api-server/README.md` 等服务级文档
- 不引入营销化或产品展示型文案风格

## 文档分工

### README

根 README 负责：

- 统一项目名称与一句话定位
- 展示核心能力与系统边界
- 给出仓库结构与服务职责
- 提供快速开始与常用入口地址
- 给出开发验证入口与文档索引
- 保留少量关键限制说明，例如不实现离线签名算法、不处理 DRM/CENC 解密

根 README 不负责：

- 完整部署排障细节
- 过多环境变量逐项解释
- 大段重复 API 示例

### DEPLOYMENT

部署文档负责：

- 说明适用环境与前置条件
- 说明配置项与示例值
- 说明启动顺序与验证方式
- 说明前后端联调方式
- 说明常见故障、安全与远程部署注意事项

部署文档不负责：

- 承担项目门面职责
- 与 README 重复大段介绍性内容

## 设计原则

### 1. 信息以当前真实行为为准

文档内容必须反映当前仓库真实状态，包括：

- 项目名称为 `DramaFlux`
- 前端开发入口为 `services/api-server/web`
- 开发态入口为 `http://127.0.0.1:5173`
- API 托管页面入口为 `http://127.0.0.1:18000/`

### 2. 绝对路径只作为示例

像 MuMu、ADB、Frida 相关路径可以保留示例，但必须明确说明：

- 这是示例路径
- 需要按本机环境调整
- 不应被读者理解为仓库默认值或强依赖

### 3. 文档层次优先于堆砌信息

使用 Markdown 层面的轻量美化增强可读性：

- 标题层级
- 徽章
- 表格
- 提示块
- Mermaid 架构图
- 清晰的“快速开始 / 深入说明”分层

不引入额外视觉素材文件。

### 4. 适合多人协作

文案语气保持工程协作型：

- 信息密度高但不臃肿
- 术语统一
- 避免个人化、本机化、临时备注式表述
- 读者默认是新加入项目的开发者

7. 常用访问地址

1. 文档用途说明
2. 环境要求
3. 依赖安装
4. MuMu / Frida / App 前置准备
5. 服务配置
6. 启动顺序

## 执行策略

采用原地重写而非小修小补：

- 直接重构根 `README.md` 的信息结构与文案
- 整理 `DEPLOYMENT.md` 中重复、过时、路径表达不清的问题
- 保留仍然正确且必要的接口与安全边界描述

## 验证标准

完成后应满足：

- 两份文档中不再出现错误项目名
- README 不再依赖本机绝对路径作为默认事实
- DEPLOYMENT 中的绝对路径都以示例形式出现
- README 与 DEPLOYMENT 分工清楚，不再互相复制大段内容
- 新同事只看 README 能知道项目是什么、有哪些服务、如何启动联调
- 需要完整部署时可以继续阅读 DEPLOYMENT，不会缺关键步骤
